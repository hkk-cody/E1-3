# Mini NPU Simulator

간단한 MAC(Multiply-Accumulate) 연산 기반 패턴 판별 시뮬레이터입니다.
콘솔에서 두 가지 모드로 동작합니다.

- 모드 1: 3x3 사용자 입력 패턴 판별
- 모드 2: `data.json` 기반 다중 크기(5x5, 13x13, 25x25) 패턴 분석

이 프로젝트는 Cross/X 필터와 입력 패턴의 유사도를 MAC 점수로 계산하고,
점수 비교 결과를 PASS/FAIL 리포트와 함께 출력합니다.

## 핵심 개념

- MAC 연산: 같은 위치 원소를 곱한 뒤 전체를 더한 값
- 라벨 정규화: `+`, `cross`, `x` 같은 입력을 내부 표준 라벨(`Cross`, `X`)로 통일
- epsilon 비교: 부동소수점 오차를 고려해 점수 차이가 매우 작으면 `UNDECIDED` 처리

## 실행 환경

- Python 3.8+
- 외부 라이브러리 불필요 (표준 라이브러리만 사용)

## 실행 방법

프로젝트 루트에서 실행:

```bash
python main.py
```

실행 후 모드 선택:

- `1`: 사용자 입력 (3x3)
- `2`: data.json 분석

## 모드별 동작

### 1) 사용자 입력 모드

- 필터 A(3x3), 필터 B(3x3), 패턴(3x3) 입력
- A/B MAC 점수 계산
- 승자 판정 (A, B, 또는 동점)
- 10회 평균 MAC 계산 시간(ms) 출력

### 2) data.json 분석 모드

- 필터 로드 상태 출력
- 패턴별 분석 수행
- 각 케이스에 대해 아래 정보를 출력
  - Cross 점수
  - X 점수
  - 판정 (`Cross`, `X`, `UNDECIDED`)
  - expected 대비 `PASS`/`FAIL`
- 성능 분석 표 출력
  - 크기(NxN)
  - 평균 시간(ms)
  - 연산 횟수(N^2)
- 최종 요약
  - 총 테스트 수 / 통과 / 실패
  - 실패 케이스와 사유

## 프로젝트 구조

```text
main.py      # 엔트리포인트, 모드 선택
mode1.py     # 사용자 입력(3x3) 분석 로직

```

## 출력 해석 가이드

- `PASS`: 판정 라벨이 expected와 일치
- `FAIL`: 검증 실패 또는 판정 불일치
- `UNDECIDED`: 두 점수 차이가 epsilon보다 작아 동점 취급
- `N/A`: 점수 계산 이전 검증 단계에서 실패해 점수를 산출하지 못한 경우

## 실패 원인 분석 (리포트)

이 프로젝트에서 FAIL은 단순 오답뿐 아니라 데이터 품질 이슈도 포함합니다.
대표적으로 패턴 키 형식 오류, 행렬 크기 불일치, 라벨 정규화 실패, 비숫자 원소가 원인입니다.
즉, FAIL은 모델 판정 실패와 데이터 스키마 실패를 함께 드러내는 진단 신호입니다.
`N/A` 출력은 점수 계산 단계에 진입하지 못했음을 의미하며,
대부분 사전 검증(`matrix_size`, `is_numeric_matrix`, 필터 존재 여부 검사)에서 걸러진 케이스입니다.
라벨은 내부적으로 `Cross`/`X`로 정규화되므로, 입력 표현 차이(`+`, `cross`, `x`)로 인한 불필요한 실패를 줄입니다.
또한 부동소수점 비교는 `epsilon` 정책으로 처리해 미세한 오차를 동점(`UNDECIDED`)으로 안정적으로 분류합니다.
예외 처리 측면에서는 MAC 계산 중 타입 이상 상황을 방어해 전체 배치 분석이 중단되지 않게 구성했습니다.
결과적으로 실패 정보는 "데이터 문제인지", "판정 정책 문제인지", "입력 품질 문제인지"를 분리해서 파악하는 데 유용합니다.

## 시간 복잡도 분석 (리포트)

MAC 연산은 NxN 행렬의 모든 원소를 한 번씩 순회하므로 시간 복잡도는 $O(N^2)$입니다.
한 번의 패턴-필터 비교에 필요한 기본 연산 수는 대략 $N^2$번의 곱셈과 $N^2$번의 덧셈입니다.
성능 분석에서는 각 크기마다 10회 반복 측정 후 평균을 사용해 단발성 노이즈를 완화했습니다.
N이 커질수록 연산량이 제곱으로 증가하므로 실행 시간도 비슷한 경향으로 증가합니다.
예를 들어 25x25는 5x5보다 한 변은 5배지만 연산 칸 수는 25배입니다.
실제 측정값은 시스템 상태 영향을 받지만, 증가 추세는 $O(N^2)$ 모델과 일치합니다.
이 특성은 필터 수가 많아질수록 총 비용이 더 커짐을 의미하며,
실제 AI 환경에서 NPU 같은 병렬 처리 장치가 필요한 이유와 직접 연결됩니다.

## CPU, GPU, NPU 정리 (간단)

- CPU: 범용 제어와 분기 처리에 강하며, 일반 애플리케이션/백엔드 로직에 적합
- GPU: 대규모 병렬 연산에 강하며, 행렬/벡터 계산과 모델 학습에 유리
- NPU: AI 추론용 MAC 경로를 하드웨어적으로 최적화해 전력 대비 성능이 높음

이 프로젝트 관점에서 핵심은 반복적인 MAC 연산입니다.
패턴 크기와 필터 수가 커질수록 연산량이 빠르게 증가하므로,
CPU 단독 처리보다 GPU/NPU 같은 병렬 처리 장치의 장점이 더 뚜렷해집니다.

실무에서는 보통 CPU + GPU/NPU 혼합 구성을 사용합니다.
CPU가 제어와 입출력을 담당하고, GPU/NPU가 대량 AI 연산을 담당하는 방식입니다.
